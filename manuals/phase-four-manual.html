What this phase delivered

PSST now has a real local browser surface instead of only a CLI and daemon socket. The new src/web-ui.ts module starts a loopback-only HTTP server on 127.0.0.1, generates a launch token, and renders the UI by calling the same Hub endpoints the CLI already uses. The browser never talks to SQLite, never reimplements selection logic, and never becomes a second Hub.

The dashboard is action-oriented rather than passive. Known Sessions are grouped into Selected Session, Available Now, Can Continue, and Past Work, and each session row can expose inspect, select or unselect, and Continue actions according to the Hub’s own liveness and continuation decision. That preserves the Phase 1 contract that visibility, selection, and controllability stay separate.

The inspect page turns a Known Session into usable local operational state. It shows PSST Session ID, working directory, Pi session file, live status, Continue action, current runtime events where available, and only the controls that the managed runtime currently says are safe. Prompt delivery and control actions such as abort and compact go back through the existing managed-runtime API rather than taking a shortcut around it.

Why this phase exists

By the end of Phase 3, PSST had the core ingredients for a browser UI but no browser-facing path: the Hub owned durable Known Session rows, selection state, Continue semantics, and managed runtime controls, yet all of that state was exposed only over a Unix domain socket. A browser cannot talk directly to that socket, and ADR 0005 explicitly postponed any web-facing endpoint until a browser client actually existed.

Phase 4 closes that gap without weakening the earlier boundaries. Instead of moving the Hub to TCP by default, or letting a web page reach into the registry directly, the implementation adds a loopback bridge that only exists when the user launches it. Token validation gates every page load, Origin validation gates mutating browser requests, and the Hub remains the source of truth for Selected Session state, Continue decisions, runtime events, and capability-gated controls.

Mental model

  1. The browser is a client, not a second Hub. Every meaningful action still passes through the stable Hub API and keeps PSST Session ID as the primary handle.
  2. The loopback bridge exists only when requested. Phase 4 does not change the default daemon transport away from HTTP JSON over a Unix socket.
  3. Selection stays a client-scope concern. The web UI uses its own scopeType=web and scopeId=local-ui selection slot instead of stealing the CLI scope.
  4. Continue stays a Hub decision. The UI offers it only when the Hub says reconnect or resume is real; it does not invent later-phase Start Fresh behavior.
  5. Controls remain capability-gated. The inspect page renders prompt input and runtime controls only when the managed adapter already says those actions are safe for the current Live Runtime.

Architecture diagram

Phase 4 adds one browser-facing seam and leaves the rest of the stack alone. The CLI starts a loopback server on demand, the bridge authenticates the browser with a launch token plus Origin checks for mutating requests, and every dashboard, inspect, select, Continue, input, control, and event read still flows back into the same Hub routes that earlier phases established.

This module is the heart of Phase 4. It starts the loopback-only server, generates the tokenized launch URL, validates Origin on mutating browser requests, renders the dashboard and inspect pages, and proxies every action back into the existing Hub API.

src/bin/psst.ts — explicit `web` launcher

The CLI gained a new web command that starts the bridge only when needed, prints the loopback origin and tokenized URL, and leaves the default daemon transport untouched.

const webUi = await startLocalWebUi({ socketPath: options.socketPath ?? defaultSocketPath() });

Phase 4 deliberately consumes the existing session list, selection, Continue, input, control, and event endpoints rather than adding a parallel browser-only transport or reaching directly into the Session Registry.

Glossary

Hub

The long-running PSST service that owns discovery, routing, the Session Registry, and client-facing selection state.

Known Session

A Pi Session that is running now or has previously been seen by the Hub.

Selected Session

The Known Session currently chosen for a given client scope, including the web UI’s own web/local-ui scope.

Session Selection

The client-side choice to point at a specific Known Session without changing the Session Registry’s durable identity model.

PSST Session ID

The Hub-owned durable handle for registry rows, selection state, and Continue actions.

Continue

The user-facing action that lets the Hub choose reconnect or resume according to runtime reachability and persisted session material.

Live Runtime

The currently running managed runtime backing a Pi Session and exposing input, event, and control capabilities.

Capability

A safe action or property the Hub believes a Known Session or Live Runtime can currently support, such as receive-input or abort.

Runtime Events

Managed-runtime events such as message updates that the Hub can expose to the inspect page when a live binding exists.

Working Directory

The local folder or repository path associated with a Pi Session and displayed in the inspect view.

Loopback web bridge

The Phase 4 browser-facing HTTP layer on 127.0.0.1 that proxies back into the existing Unix-socket Hub API.

Origin validation

The browser-side safety check that rejects mutating web requests unless they originate from the launched loopback UI origin.

What's next

Phase 5 should add Telegram as another client adapter over the same Hub semantics rather than inventing a second routing model. The prerequisite gate is to preserve what Phase 4 just proved: the Hub remains the only source of truth for Known Session identity, selection, Continue behavior, and runtime capabilities, while remote clients must consume those semantics instead of re-deriving them from transport-specific assumptions.
