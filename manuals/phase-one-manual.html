What this phase delivered

PSST is now an executable local-first Hub rather than a collection of ADRs and terminology notes. The repository contains a Node 22 and TypeScript implementation that starts a foreground Hub daemon, persists Known Sessions in SQLite with WAL enabled, and exposes the first stable control surface over HTTP and JSON on a Unix domain socket.

The heart of the phase is the Session Registry. The registry owns the primary PSST Session ID, accepts registrations from Pi-facing surfaces, applies the required duplicate-match priority of PSST Session ID then Pi session file then Pi Session ID, and keeps client-side Session Selection separate from liveness and controllability. This is the phase that turns those distinctions from prose in CONTEXT.md into code and tests.

The CLI is intentionally thin. It talks to the same socket API the daemon serves, so every successful manual exercise and every end-to-end test is proving the public contract rather than a private helper path. That matters because later phases need to build on the same contract from a managed Pi adapter, a web UI, and a Telegram adapter without redefining what the Hub means.

Why this phase exists

Before this phase, PSST had a strong domain model but no runtime to enforce it. There was no daemon, no registry, no persisted selection state, no socket transport, and no executable proof that the product framing had actually escaped the planning documents. Everything important was still a promise.

Phase 1 establishes the minimum durable contract that the rest of the rewrite depends on. Once the Hub can remember Known Sessions, classify them as live, stale, or dead, choose a Continue action, and let a client select work safely, the later phases can focus on richer control surfaces instead of renegotiating basic identity and lifecycle semantics.

Mental model

  1. PSST owns the primary identity. Every Known Session gets a PSST Session ID that survives duplicate registration and client selection.
  2. Visibility, selection, liveness, and controllability are separate concerns. A session can exist in the registry without being selected, live, or able to receive input.
  3. Duplicate matching is strict and ordered. PSST Session ID wins, then Pi session file, then Pi Session ID. Working directory and machine are explicitly not enough to collapse sessions.
  4. Continue is a decision, not a synonym. The daemon computes whether the next user-facing action is reconnect, resume, or start-fresh from liveness and persisted material.
  5. The CLI is just a client. If a feature matters, it is represented by the socket API first and the CLI second.

Architecture diagram

The implementation is intentionally shallow in process topology and deep in registry semantics. One daemon owns the SQLite file, one Unix socket exposes the API, and every client goes through that same contract.

npm install

Expected outcome — dependencies install successfully and the repo is ready to run tests and CLI commands.

What this proves — the Phase 1 repo is a real Node 22 package, not a documentation-only skeleton.

Exercise 2 — prove the contract with the full suite

Command

npm test && npm run typecheck && npm run build

Expected outcome — 13 tests pass, TypeScript emits no errors, and the project builds cleanly.

What this proves — the registry, daemon, lifecycle, and CLI surfaces are all exercised and buildable from a clean checkout.

Exercise 3 — start a real Hub daemon

Command

ROOT="$(mktemp -d)" && \
node --import tsx src/bin/psst.ts daemon \
  --data-dir "$ROOT/data" \
  --runtime-dir "$ROOT/run"

Expected outcome — the command prints JSON containing ok: true and a socket path, then stays in the foreground until you stop it.

What this proves — the Hub can create its runtime directory, acquire its lock, bind the Unix socket, and enter service mode.

Exercise 4 — register Known Session metadata through the CLI

Command

ROOT="<same path from exercise 3>" && \
node --import tsx src/bin/psst.ts --socket "$ROOT/run/hub.sock" session register \
  --name "Manual demo" \
  --working-directory "$PWD" \
  --pi-session-file /tmp/manual-demo.jsonl \
  --pi-session-id pi-manual

Expected outcome — JSON prints a generated psst_... identifier plus live status, capability flags, and Continue action.

What this proves — PSST owns session identity and can persist a Known Session independently of the eventual Pi control plane.

Exercise 5 — inspect the Session Registry state

Command

ROOT="<same path>" && \
node --import tsx src/bin/psst.ts --socket "$ROOT/run/hub.sock" session list

Expected outcome — the session list shows one Known Session with its PSST Session ID and persisted metadata.

What this proves — the daemon is reading the SQLite registry, not holding the session only in memory.

Exercise 6 — select current work

Command

ROOT="<same path>" && \
SESSION_ID="$(node --import tsx src/bin/psst.ts --socket "$ROOT/run/hub.sock" session list | jq -r '.sessions[0].psstSessionId')" && \
node --import tsx src/bin/psst.ts --socket "$ROOT/run/hub.sock" select "$SESSION_ID" && \
node --import tsx src/bin/psst.ts --socket "$ROOT/run/hub.sock" current

Expected outcome — the select call marks the session selected, and the current call returns the same PSST Session ID for client scope cli/default.

What this proves — Session Selection is persisted separately from session registration and can be resolved without guessing.

Exercise 7 — ask the Hub how to Continue

Command

ROOT="<same path>" && \
SESSION_ID="$(node --import tsx src/bin/psst.ts --socket "$ROOT/run/hub.sock" session list | jq -r '.sessions[0].psstSessionId')" && \
node --import tsx src/bin/psst.ts --socket "$ROOT/run/hub.sock" continue "$SESSION_ID"

Expected outcome — JSON returns resume for a persisted Pi session file or start-fresh when no persisted material exists.

What this proves — Continue is already a first-class user-facing decision, even before Phase 2 adds actual runtime control.

Key components

src/daemon.ts — foreground daemon, socket listener, and route wiring

The daemon creates the runtime directory, acquires a lock, removes stale sockets, boots the registry, and binds the HTTP server.

const lock = await acquireRuntimeLock(lockPath);
const registry = await SessionRegistry.open(options.dataDirectory, {
  deadAfterMs: options.deadAfterMs,
  staleAfterMs: options.staleAfterMs,
});
await listenOnSocket(server, socketPath);
src/registry.ts — SQLite schema, migrations, duplicate matching, and selection persistence

This module encodes the Phase 1 contract directly: WAL mode, migration bootstrap, ordered duplicate detection, lifecycle events, and selection state.

if (input.psstSessionId) return byPsstId;
if (input.piSessionFile) return bySessionFile;
if (input.piSessionId) return byPiSessionId;
return null;
src/client.ts — raw Unix-socket JSON client

The CLI uses a single request helper instead of duplicating socket request code per command, which keeps the public contract centered on HTTP rather than ad hoc process calls.

const req = request({
  socketPath: this.socketPath,
  path,
  method,
  headers
});
src/bin/psst.ts — CLI entrypoint

The CLI stays intentionally narrow: parse the command, call the socket client, emit JSON, and let the daemon own behavior.

if (rootCommand === "select") {
  writeJson(await client.post("/v1/selections", {
    scopeId: options.scopeId,
    scopeType: options.scopeType,
    sessionId,
  }));
}

Glossary

Hub

The long-running PSST service that owns discovery, routing, the Session Registry, and client-facing selection state.

Known Session

Pi work that is running now or has previously been seen by the Hub.

PSST Session ID

The stable identifier owned by the Hub and used as the primary handle for registry rows and selections.

Pi Session ID

A technical identifier from Pi that is useful for deduplication hints but is not PSST’s primary identity.

Session Selection

A client-side choice to interact with a specific Known Session.

Selected Session

The Known Session currently pointed to by a client scope such as cli/default.

Last Seen

The timestamp of the most recent registration or heartbeat used to classify a session as live, stale, or dead.

Continue

The user-facing action for getting back to work, choosing reconnect, resume, or start-fresh.

Resume

Continue a Known Session from persisted Pi material after the original runtime has gone away.

Reconnect

Restore a control path to an existing Live Runtime.

Stale Session

A Known Session that has not been seen recently but may still be recoverable.

Dead Session

A Known Session the Hub believes is no longer reachable.

What's next

Phase 2 is the next gate. It should keep the Hub contract stable and add managed Pi runtime control through the official SDK first and RPC second. The prerequisite is to preserve the invariants proven here: PSST-owned identity must remain primary, selection must stay distinct from controllability, and Continue must keep meaning “the Hub chooses reconnect or resume” rather than becoming an unstructured catch-all action.
