What this phase delivered

PSST can now learn about Pi Sessions that were launched directly by the user inside Pi rather than only through the managed adapter path. The new psst-pi-extension module performs a narrow session_start-time registration against the existing Hub API, carries forward any prior PSST Session ID it finds in the session JSONL, and leaves the Hub in charge of Known Session identity, selection, and Continue semantics.

The extension enriches the registry only with metadata that belongs in a Known Session row: working directory, Pi session file, Pi Session ID, Pi Session Name, model, thinking level, and a deliberately safe capability subset. It refuses to advertise control powers such as input delivery or abort just because the extension is loaded. That preserves the ADR and glossary boundary between visibility and controllability.

Two more behaviors make the phase operational rather than aspirational. Successful registration writes a psst.linkage entry back into the Pi session JSONL using an appendEntry-style seam, and the module exposes a TUI status message type plus renderer so Pi can explain whether the Hub linked the session, carried the identity into a replacement runtime, or stayed offline without breaking the user’s work.

Why this phase exists

Phase 2 gave PSST a managed adapter for runtimes that the Hub starts or resumes itself, but it still left a practical blind spot: a user can open Pi directly, start work, and never pass through the managed launcher at all. Without a narrow self-registration bridge, those live Pi Sessions stay invisible to the Hub until some later manual step recreates their metadata.

Phase 3 closes that gap without violating the Phase 1 or Phase 2 contracts. The extension is optional, best-effort, and metadata-oriented. When the Hub is reachable it self-registers and persists the resulting PSST linkage; when the Hub is absent it surfaces an offline status and leaves the Pi Session local. This gives later UI and Telegram phases richer visibility without confusing a Pi plugin with the Hub itself.

Mental model

  1. The Hub still owns Known Session identity. The extension may reuse a PSST Session ID it discovers, but it does not mint a second identity system.
  2. Self-registration is optional and best-effort. If the Hub is absent, the Pi Session must continue locally and surface an offline status instead of failing startup.
  3. psst.linkage is the carry-forward seam. Persisting it into session JSONL lets replacement session_start flows keep the same Known Session rather than creating duplicates.
  4. Metadata enrichment is additive, not controlling. Working directory, session file, model, thinking level, and safe capability hints can be projected into the registry without implying that the extension can steer or abort work.
  5. Phase 2 remains the primary control path. Phase 3 gives PSST richer visibility and user guidance, not a rival runtime-control plane.

Architecture diagram

Phase 3 inserts a small bridge inside the Pi Session. The Pi extension reads any prior linkage from session JSONL, registers with the same Hub API the CLI already uses, persists the resulting PSST Session ID back into JSONL, and renders a status message inside Pi so the user can see what PSST learned without confusing the extension for the Hub.

Glossary

Hub

The long-running PSST service that owns discovery, routing, the Session Registry, and client-facing selection state.

Known Session

A Pi Session that is running now or has previously been seen by the Hub.

Pi extension

The optional Phase 3 bridge loaded inside a Pi Session that can self-register with the Hub and surface PSST status in the TUI.

Session Registry

The Hub’s durable store of Known Sessions and their selected metadata.

PSST Session ID

The Hub-owned durable identity for registry, selection, and Continue actions.

Pi Session ID

An external identity hint that may change across replacement and is not the primary PSST handle.

Pi Session Name

The human-readable name of a Pi Session, stored in PSST as the session name rather than as a separate display-name abstraction.

Session Metadata

Descriptive facts about a Pi Session such as working directory, session file, model, thinking level, and capability hints.

Continue

The user-facing action that chooses reconnect, resume, or Start Fresh according to runtime reachability and persisted material.

Replacement runtime

A new Pi runtime that carries forward the same Known Session identity after the previous runtime or session file is replaced.

TUI status message

The Phase 3 message shown inside Pi to explain whether PSST linked the session, stayed offline, or carried the same identity into a replacement.

appendEntry linkage

The persisted psst.linkage JSONL record that lets replacement session_start flows reuse the same PSST Session ID.
