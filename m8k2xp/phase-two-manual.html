What this phase delivered

PSST now has a managed Pi control layer that sits on top of the Phase 1 Hub rather than replacing it. The Hub still owns Known Session identity, selection state, and durable registry semantics, but it can now also attach a Live Runtime, keep that runtime distinct from the persisted Session Registry row, and expose runtime actions through the existing Unix-socket HTTP surface.

The main addition is ManagedPiRuntimeService. It tracks active runtime bindings in memory, routes Continue through the required decision tree, reattaches event subscriptions after runtime replacement, and updates the registry only with the metadata that belongs in a Known Session: current Pi session binding, capabilities, and liveness. That keeps the ADR boundaries intact. Runtime reachability remains separate from persisted session identity.

The phase also adds explicit backend modules instead of hiding Pi integration behind ad hoc helpers. SdkPiAdapter maps official session APIs such as prompt, steer, follow-up, compact, and new/switch/fork/clone; RpcPiAdapter speaks JSONL over a spawned pi --mode rpc process and consumes runtime events. The factory module prefers the SDK path when a launcher is available and falls back to RPC otherwise.

Why this phase exists

Phase 1 deliberately stopped at registry and Continue semantics. That was enough to prove that PSST could remember Known Sessions and classify them without muddling identity, selection, and liveness, but it still left one important gap: Continue did not actually reconnect or resume work. It only returned the next recommended action.

Phase 2 closes that gap without breaking the earlier contracts. The Hub now chooses reconnect when it already owns a reachable Live Runtime, resume when persisted Pi material exists but the runtime is gone, and Start Fresh when neither path exists. It also adds the public routes needed for real control: input delivery, runtime controls, and event inspection. This is the contract that later self-registration, UI, and Telegram work will build on.

Mental model

  1. A Known Session is still the durable unit. Live Runtime tracking is kept outside the Session Registry and only projects safe metadata back into it.
  2. Continue is executable but still constrained: reconnect first, resume second, Start Fresh last. The Hub chooses; clients do not hard-code that tree.
  3. Runtime replacement is normal. New, switch, fork, and clone may invalidate old session-bound objects, so event subscriptions must move with the new binding.
  4. The input/control/event API is an extension of the Phase 1 socket contract, not a second transport surface. CLI and future adapters stay clients of the same HTTP API.
  5. SDK-first means preference, not lock-in. When an official SDK launcher is available, PSST uses it. When stronger isolation or SDK absence calls for it, the RPC backend is the fallback.

Key components

src/managed-pi.ts — runtime tracking, Continue execution, and runtime replacement

This module is the heart of the phase. It keeps runtime bindings out of the registry, executes the Continue tree, forwards input/control operations, and re-subscribes on replacement.

if (binding) return { action: "reconnect", session: ... };
if (!session.piSessionFile && !session.capabilities.canResume) return { action: "start-fresh", session };
const handle = await this.adapter.resumeSession(...);
await this.attach(psstSessionId, handle);
src/pi-sdk-adapter.ts — SDK backend wrapper

The SDK backend is intentionally thin. It maps public session methods into the managed-handle interface and turns replacement sessions into new handles instead of exposing SDK objects upstream.

case "follow-up":
  return session.followUp(text);

return {
  replacement: result.session ? wrapSdkSession(result.session) : undefined,
  state: result.state,
};
src/pi-rpc-adapter.ts — JSONL RPC backend

This adapter spawns pi --mode rpc --name "PSST:<session-name>", tracks pending requests by id, updates mutable runtime state from get_state and command responses, and forwards event lines to subscribers.

const id = randomUUID();
child.stdin.write(`${JSON.stringify({ command, id, ...payload })}\n`);
if (message.id && pending.has(message.id)) { ... }
if (message.eventType && message.occurredAt) { ... }
src/daemon.ts — HTTP surface extension

Phase 2 does not add a new server. It extends the existing daemon with the new runtime routes and keeps them on the same Unix-socket JSON API.

POST /v1/sessions/:id/input
POST /v1/sessions/:id/control/(abort|compact|new|switch|fork|clone)
GET  /v1/sessions/:id/events
POST /v1/sessions/:id/continue

Glossary

Hub

The long-running PSST service that owns discovery, routing, the Session Registry, and client-facing selection state.

Known Session

A Pi Session that is running now or has previously been seen by the Hub.

Live Runtime

The currently reachable Pi control path attached to a Known Session.

Managed Pi adapter

The Phase 2 layer that turns Pi runtimes and session controls into a Hub-owned contract.

PSST Session ID

The Hub-owned durable identity for registry, selection, and Continue actions.

Pi Session ID

An external identity hint that may change across replacement and is not the primary PSST handle.

Continue

The user-facing action that chooses reconnect, resume, or Start Fresh according to runtime reachability and persisted material.

Reconnect

Restore control to an already reachable Live Runtime.

Resume

Create a replacement runtime from persisted Pi material when the old runtime is gone.

Start Fresh

Create or request a new Pi Session when there is no reachable runtime and nothing resumable.

SDK backend

The preferred official integration path that exposes prompt, steer, follow-up, and control operations through session objects.

RPC backend

The fallback backend that spawns Pi in RPC mode and exchanges JSONL commands and events.

What's next

Phase 3 should build on this runtime layer without weakening it. The next step is optional Pi extension self-registration so Pi-launched sessions can announce themselves safely while the Hub keeps PSST-owned identity primary. The gate before that work begins is simple: keep the new managed runtime service as the place where Live Runtime logic lives, keep the registry durable and runtime-agnostic, and avoid smuggling later UI or Telegram concerns back into the core transport and selection contract.
